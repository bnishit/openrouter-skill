Why This Exists

OpenRouter moves faster than app code usually does.

New models appear quickly, providers differ, and pricing shifts underneath existing flows. Teams end up redoing the same integration steps every time — choosing a model, comparing providers, checking free options, adding image generation, auditing generation cost, handling tool loops, and wiring up multimodal inputs. This skill bundles those recurring patterns into one installable layer so agents can reason about model choices, provider tradeoffs, and cost inspection without rebuilding from scratch.

First Principles

Why this matters before anyone talks about prompts.

01

Model markets change quickly

If you care about using the newest useful model instead of hardcoding last month's choice, you need model discovery, provider awareness, and safe routing built into the workflow.

02

Compatibility is the reason teams choose OpenRouter

One OpenAI-compatible layer makes it easier to add a new text, vision, image-generation, or file-ready model without redesigning the whole integration every time.

03

Cost matters after the request, not just before it

Estimating from pricing tables is not enough. Real teams need exact generation lookup for billing, debugging, and "why was this expensive?" questions later.

04

Scaffolding should not be rebuilt every project

Starter routes, proxy setup, streaming config, and multimodal request shapes are the same across projects. Rebuilding them is wasted engineering time.

05

Production rules are learned the hard way — once

Fallback behavior, tool-loop safety, JSON validation, and cost logging are operational rules that teams discover through incidents. A skill should encode them upfront.

06

Agents need structured knowledge, not just API docs

An API reference tells you what endpoints exist. A skill tells an agent when to use them, what tradeoffs to consider, and how to handle the edge cases that docs don't cover.

How It Works

Real situations, real outcomes.

01

Shipping fast on a new model

A new image-capable model just appeared. You want image generation without rewriting the app or hardcoding another provider-specific path.
The skill fetches the live catalog, filters by modality, scaffolds the route, and keeps the integration OpenRouter-compatible — no vendor lock-in.

When the question is "can I keep adapting?" not just "can I call a model?"

02

Cost clarity after the fact

Yesterday's OCR flow cost more than expected. You need exact token accounting, not estimates from pricing tables.
The skill looks up the generation by ID, returns exact cost, provider, and token fields — wired into logs or admin UI.

When cost is an operational question, not just an architecture slide.

03

Low-cost and fallback-conscious builds

You want a workflow that starts cheap, can use free models when possible, and still has provider-aware fallbacks when quality matters.
The skill filters zero-priced models, lists providers, preserves routing controls, and scaffolds the endpoints your app needs.

When the real job is balancing capability, price, and fallback behavior at once.

04

One path for text, vision, and images

You don't want one SDK for chat, another for images, and a third for file workflows. You want one integration that keeps up.
The skill keeps the app on one OpenRouter-compatible layer across all modalities — same discovery, routing, and cost-audit patterns.

When the win is operational simplicity, not just one successful request.

05

Icons and OG images without custom workflows

You need app icons and OG images but don't want to rebuild prompts, preview handling, file naming, and asset metadata every time.
The skill uses shipped image-asset presets, previews returned image data immediately, and persists with generation ID, model, and prompt metadata.

When image generation is part of a product workflow, not just a one-off demo.

06

Production rules without rediscovering them

The feature is working, but you don't want to remember all the small rules for fallbacks, tool loops, JSON validation, and cost logging every time.
The skill applies shipped best-practice playbooks so implementation covers the operational rules teams usually learn the hard way.

When the gap is not API syntax, but repeatable engineering judgment.
